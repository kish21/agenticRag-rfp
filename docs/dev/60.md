# #60 (P1.9) — Human feedback capture → few-shot bank

**Status:** BUILT (PR pending) · **Date:** 2026-06-07 · **Touches the Evaluation Agent prompt → benchmark re-run mandatory (done, unchanged).**

## What & why

When a human reviewer overruled the AI, the correction was thrown away — the next run repeated the mistake. P1.9 closes the loop: a reviewer corrects a specific **criterion score** or **mandatory check** decision (with a reason); the correction is stored org-scoped and injected back into the Evaluation Agent's prompt for that same criterion/check on future runs. The platform learns from its own reviewers. The Critic still runs after every agent — examples **guide**, they never bypass guardrails.

### Key finding (drift the old build note got wrong)
The existing override path (`app/domain/override.py`, `POST /{run_id}/override`) records **whole-vendor** decisions only (`{vendor_id, reason}`) — not which criterion/check was wrong or the corrected value. The dev BACKLOG's real goal is *"when an evaluator overrides an AI **score**"* — i.e. criterion/check grain. So #60 adds a finer capture path (`evaluation_corrections`) that **also** writes an `AuditOverride` for the audit trail (Component Contract #7). The vendor-level `/override` endpoint is untouched.

### Decisions (confirmed with user)
- Correct at **per-criterion / per-check** grain (not whole-vendor).
- Few-shot applies to **both** mandatory checks and criterion scores.
- **Full-stack** in one session (incl. the reviewer screen).

**Exit criteria (all met + verified):**
- [x] Reviewer can correct one criterion score or one check, with a reason → stored + fed back. (`/correct`, few-shot injection)

- [x] Critic still runs; examples are non-authoritative guidance (hard rules in the prompt; guardrail paths untouched).

**Module-interaction map (typed contracts in/out):**
```
Reviewer UI ──POST /correct {target_type,target_id,vendor_id,corrected_value,reason}──► evaluation_routes.submit_correction
  → validate (target enum, score 0-10 / decision enum, reason≥20, injection-scan) · require_run_access · require_admin_role
  → fact_store.save_evaluation_correction(EvaluationCorrection)  → evaluation_corrections (RLS, org-scoped)
  → override.save_override(AuditOverride)                         → audit_overrides  (Contract #7)
  → audit("correction.submitted")
Evaluation Agent ──build_few_shot_block(org,target_type,target_id,name)──► few_shot service
  → fact_store.get_evaluation_corrections(org_id, target_type, target_id, limit)   (RLS + org filter)
  → injection-scan each → render via prompts/evaluation/few_shot_examples.yaml
  → injected into evaluate_check / score_criterion prompt (after facts, before the JSON instruction)
```

**Typed contract:** `EvaluationCorrection` (`app/schemas/schema_evaluation.py`) — `reason` ≥20 chars (validator), `target_type ∈ {criterion, check}`, open `original_value`/`corrected_value` dicts (criterion → `{raw_score:int}`, check → `{decision:str}`).

## Security DoD (verified)
- **Tenant isolation:** RLS policy `rls_evaluation_corrections` (FORCEd via the dynamic block) + explicit `org_id` filter; cross-org read proven impossible (`test_tenant_isolation_rls.py::test_evaluation_corrections_isolation`). A tenant only learns from its own corrections.
- **AuthZ:** `require_run_access` + `require_admin_role` on `/correct`; org-scoped run lookup 404s cross-org before authz. `org_id` is server-derived from the JWT, never the body.
- **Input validation:** target enum; score must be int 0-10 (bool excluded — `isinstance(int)` but not `bool`); decision ∈ `ComplianceStatus`; vendor must belong to the run (404); reason ≥20 at route + model + DB CHECK.
- **Prompt-injection (OWASP LLM01):** reason scanned with `app/validators/injection.py` at **submit** (fail-CLOSED 400) and re-scanned at **injection time** (skip). SQL fully parameterized; prompt YAML via `yaml.safe_load`; React auto-escapes the corrections list.
- **Fail-safe:** any bank read/render error → empty block, evaluation continues; Critic unaffected.
