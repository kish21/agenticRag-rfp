## Context — what was already done before this ticket

Issue #55 ("Multi-user RBAC within org") is **~85% pre-built**. A deep verification
(commit `fb5336d` #46 + Phase 9 visibility + tenant-isolation #0011) found the core
already shipped and tested:

| #55 ask | Pre-existing state |
|---|---|
| Role-based visibility (default-deny within org) | ✅ `runs_visible_to()` SQL fn + `app/domain/visibility.py` |
| Postgres RLS policies (actually enforced) | ✅ FORCE RLS on every table, non-superuser `platform_app` role (migration 0011) |
| `access_audit_log` table | ✅ schema + RLS + fire-and-forget `log_access()` |
| owner / dept-member / approver / admin roles | ✅ `created_by_email` / `user_departments` / `approval_assignments` / `company_admin`+`department_admin` |
| Endpoint enforcement (403 on others' runs) | ✅ `require_run_access` on every per-run endpoint |

**The one genuine gap** named in the issue: there is **no `auditor` role** — a read-only
compliance persona who can review the org's *audit trail* (who-accessed-what + override /
state-change events) **without** being able to run evaluations or see negotiation content.
That is what this ticket builds. (CFO / approver / "owner" from the issue text are already
covered by the approver-assignment + ownership mechanisms; adding them as first-class JWT
roles was rejected as domain over-fitting — see [[generic_platform_no_domain_special_case]].)

## Per-feature contract

### Exit criteria (testable)
1. `auditor` is a valid JWT role: `create_access_token(role="auditor")` succeeds and
   `decode_token` accepts it; `users.role` CHECK allows it (migration applied).
2. An `auditor` token is **blocked (403 / empty)** from every run-content path:
   `/start` (403), `/list` (empty), `/{id}/setup|results|decision|cost|audit|export`,
   `/confirm`, `/override`, `/re-evaluate`, `/cancel`, `DELETE /{id}`, collaborators,
   `/{id}/status` + `/{id}/stream` (SSE) — all 403 via default-deny `require_run_access`.
3. An `auditor` token **can read** the new org-wide audit endpoints:
   `GET /api/v1/audit/access-log` and `GET /api/v1/audit/events` (200, org-scoped).
4. A non-audit-read role (e.g. `department_user`) gets **403** on those two endpoints.
5. Which roles may read the audit trail is **config-driven** (`product.yaml rbac.audit_read_roles`),
   not hardcoded in the route.
6. Auditor is **not** in `write_roles`, so RFP/setup/admin-attribution writes stay 403.
7. Tests pass + full non-live suite green; `/security-review` clean.

### Module-interaction map (typed contracts)

## Within-org leak fixed in passing (#55 core)
The SSE progress endpoints `GET /{id}/status` and `GET /{id}/stream` previously gated
**only** on org membership (`_db_get_run(run_id, user.org_id)`), not `require_run_access`
— so any same-org user (incl. an auditor) could subscribe to another run's
`agent_events` + status. That is the exact "within-org, not cross-org" leak #55 was
filed to close, and every other per-run endpoint already calls `require_run_access`.
Both now enforce it (+ `403` documented in their OpenAPI responses). Surfaced by the
`/security-review` sub-agent; the earlier code map had mislabelled these as already-guarded.

## Design decisions
- **Auditor is NOT a wide role.** `_WIDE_ROLES` (visibility) stays `{platform_admin, company_admin}`.
  Auditor matches no run predicate → default-deny already gives 403 on every per-run endpoint.
  Only `/start` (creates) and `/list` `scope=auto` else-branch (returns whole org) bypass
  visibility, so those two get an explicit guard.
- **Least privilege:** auditor sees the *audit trail metadata* (run_id, accessor, action, event
  type), never run results / decisions / negotiation content.
- **Config-driven** audit-read roles so product can widen/narrow without an engineering change.
- **No new write surface.** Auditor stays out of `write_roles`; audit endpoints are read-only SELECTs.
- **One launch gate, not scattered literals.** Every run-launching endpoint
  (`/start`, `/confirm`, `/re-evaluate`, `/rerun`) now calls the single config-driven
  `rbac.require_write_role` (keys off `write_roles`), so "a read-only role cannot launch
  evaluations" holds even for a future read-only role that has run *visibility*. The
  `/list` read-only exclusion is likewise config-driven (write_roles), not a role literal.
  (Code-review finding: the launch guard was initially only on `/start`.)
