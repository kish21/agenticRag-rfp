## Context (why this is being built)

Issue #119 is the only open **P0** and the top enterprise-sales blocker: an EU/UK enterprise's
legal/security team will not sign until we can answer "yes" to *"can you honour a data-deletion
request and erase a customer on offboarding?"*. It is used rarely (a sales/compliance checkbox),
not daily.

### Key framing (decided with the owner)
GDPR's "right to be forgotten" is about **people's personal data**, not a company throwing away its
own business records. RFP responses and decisions are business/audit records the customer *wants*
and is *legally allowed* to keep (GDPR Art. 17(3) exempts data held under a legal-retention
obligation — which is why `product.yaml` keeps decisions 7 years). So the real feature is **two
distinct modes**:

- **Mode B — Tenant erasure (offboarding):** when a customer *leaves*, wipe their entire tenant.
  This is what issue #119 literally describes ("erase all their data, one API call"). **← this doc.**
- **Mode A — Subject anonymization (an individual asks):** scrub one person's personal identifiers
  (name/email) but KEEP the business/audit records, now linked to a "redacted" subject. **← next
  subtask, scoped below, not built here.**

## Scope of THIS subtask (Mode B — tenant erasure)

One admin-gated, audited, deliberate operation that erases everything we directly control for one
org: PostgreSQL rows → Qdrant vectors → on-disk uploaded files → in-memory caches, then writes a
single **retained, anonymized erasure receipt** as proof it happened (org_id + who-requested +
timestamp + per-table counts; no PII). The org's own audit/decision history goes with the tenant
(they're leaving) — but the receipt survives, preserving the "append-only audit proof" principle.

## Approach & files

**New — `app/domain/org_erasure.py`** (orchestrator):
`erase_org(org_id, *, requested_by, reason) -> ErasureReceipt`. Order:
1. Qdrant: call existing `delete_org_data(org_id)` — [app/retrieval/qdrant.py](../../app/retrieval/qdrant.py)
   `:336` (already purpose-built for "delete this customer"; returns `(points, collection_dropped)`).
2. PostgreSQL: call new `purge_org_postgres(org_id)` (below).
3. Disk: remove drop folders for the org's RFPs — reuse `_drops_root()` / `provision_drop_folder`
   path logic in [app/api/rfp_routes.py](../../app/api/rfp_routes.py)`:53-72` (iterate the org's
   `rfps`, `shutil.rmtree({drops_root}/{rfp_id})`).
4. Caches: `invalidate_org_settings(org_id)` ([app/domain/org_settings.py](../../app/domain/org_settings.py)`:98`);
   clear per-run cost accumulators for the org's runs (`app/infra/cost_tracker.py clear_run_cost`).
5. Write the retained receipt via `audit(... event_type="org.erased" ...)` BEFORE the org row is
   gone (`audit_log.org_id` has **no FK** to `organisations` — confirmed in `schema.sql` — so the
   receipt survives a full org delete).

**New — `app/db/fact_store.py :: purge_org_postgres(org_id) -> dict[str,int]`**:
FK-safe ordered deletes returning per-table counts. Mirror the proven ordering in the `delete_run`
handler ([app/api/evaluation_routes.py](../../app/api/evaluation_routes.py)) but org-wide.
Children→parent: approvals, decisions, audit_overrides → extracted_* (6 tables) → vendor_documents,
evaluation_runs, evaluation_setups → rfps/invited_vendors/ingestion_jobs/event_log → agent_registry,
org templates, tenant_modules, tenant_billing, org_settings(+audit), auth_sessions/tokens, users →
finally `organisations`. Use `get_admin_engine()` (RLS-exempt; cross-cutting system op) in ONE
transaction. Validate the exact table list + order against [app/db/schema.sql](../../app/db/schema.sql)
at implementation time (cascade clauses mean several children auto-delete).

**New endpoint — `app/api/admin_routes.py :: DELETE /api/v1/org/{org_id}/data`**:
- RBAC: `require_role("platform_admin", "company_admin")` + ownership check (company_admin may only
  erase **their own** org; platform_admin any). Mirror the audited-destructive pattern of
  `purge_llm_cache` in the same file.
- Safety: require an explicit confirmation field (e.g. body `confirm_org_name` must match) — no bare
  "delete everything"; refuse (409) if the org has runs currently `running`.
- Pre-op audit `org.erasure_requested`, then call `erase_org(...)`, return the receipt (counts).

**Config (no hardcoding) — `app/config/product.yaml` + `loader.py`**: small `gdpr` block, e.g.
`keep_erasure_receipt: true`, `block_if_runs_in_flight: true`. Defaulted so older yaml still loads.

**Audit event types — `app/infra/audit.py`**: register `org.erasure_requested`, `org.erased`.

## Out of scope here (documented follow-up subtasks — also add to BACKLOG.md)
- **Mode A — individual subject anonymization** (scrub one person's `users.email`/`full_name` +
  identifier references in `created_by_email`/`actor`/`accessed_by`/`invited_by`, keeping business
  records). This is the everyday GDPR request and a clean separate session.
- **`llm_response_cache`** is tenant-blind (no org_id) — cannot selectively erase. Residual-risk
  note now; future per-org cache keying.
- **LangSmith traces** — external API deletion, best-effort follow-up.
- **Vendor-contact PII inside document text / extracted facts** — covered by the tenant wipe here;
  for Mode A it needs content redaction (hard) — note as limitation.
