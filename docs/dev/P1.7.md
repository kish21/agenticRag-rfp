# P1.7 — Self-consistency voting for borderline compliance checks

**Status:** ✅ BUILT (2026-06-06). All 7 exit criteria verified — 8 unit/integration tests
green + benchmark unchanged (grounding 1.0, 0 fabricated, 0 failures).
**Owner decisions (2026-06-06):** ON by default · no-majority → fail-safe `insufficient_evidence`.

**One deliberate deviation from the pseudocode below:** the baseline call keeps today's
**auto-derived** seed (`call_llm(messages, temperature=0.0, ...)` — no explicit `seed=`)
rather than `seed=0`. The auto-derived seed (sha256 of messages) is *equally* deterministic
and a stable audit baseline, and keeping the baseline call byte-for-byte identical to the
prior single call is what makes exit-criteria #1 (no behaviour/cost change off-band) and #7
(benchmark unchanged) hold tightest. Only the **resamples** take explicit seeds (`seed=1..N-1`)
+ `cfg.temperature`; the LLM cache keys on `(temperature, seed)`, so the samples are genuinely
distinct yet reproducible. See [evaluation.py `_decide_check_with_voting`](../../app/agents/evaluation.py).

---

## Context / problem

Each mandatory compliance check (e.g. "holds ISO 27001", "carries ≥ £5M PI insurance")
is decided by a **single** LLM call at [evaluation.py:253](../../app/agents/evaluation.py#L253),
temperature 0. For clear-cut checks the model is stable. For **borderline** checks
(ambiguous / half-stated evidence) the same prompt can yield different verdicts on
different runs — a fragile single roll of the dice on the highest-stakes decision in
the pipeline (a flaky borderline call can wrongly reject a good vendor or shortlist a
non-compliant one).

**Self-consistency voting:** for borderline checks only, sample the same decision N
times and take the majority. Clear-cut checks stay single-call (no added cost).

Not currently built: the evaluation agent calls the LLM exactly once per check; no
voting / majority / borderline-band logic exists anywhere. (The critic-retry loop is
retry-with-feedback, NOT voting; `_llm_verify_threshold` is a per-chunk fallback, NOT
voting — do not conflate.)

---

## Per-feature contract

### Exit criteria (testable)
1. A check whose **primary confidence is outside** `[confidence_min, confidence_max]`
   makes exactly **one** LLM call (no behaviour/cost change). *(unit, mocked)*
2. A check whose **primary confidence is inside** the band triggers `samples` total
   calls and the returned `decision` is the **majority** of the sampled decisions.
   *(unit, mocked: 2×pass+1×fail → pass)*
3. **No-majority** (e.g. 1 pass / 1 fail / 1 insufficient) → `insufficient_evidence`,
   regardless of which sample was most confident. *(unit, mocked)*
4. Reported `confidence` after a vote = **agreement ratio** (winning votes / samples),
   and a **vote breakdown** is recorded for audit. *(unit)*
5. `enabled: false` → identical behaviour to today (single call). *(unit)*
6. The contradiction-override (E3.b) and chunk-fallback logic that follow the primary
   call are **unchanged** and still apply to the voted decision. *(regression)*
7. Benchmark unchanged: grounding 1.0, 0 fabricated, 0 failures; no spurious flips on
   the synthetic set. *(live benchmark)*

### Module-interaction map
- **In:** `_evaluate_mandatory_check(...)` builds `messages` exactly as today.
- **New seam:** a helper `_decide_check_with_voting(messages, cfg) -> (parsed: dict, votes: dict)`
  replaces the single `call_llm` + `json.loads` at lines 253–257. It returns ONE
  representative `parsed` dict (so all downstream logic — contradiction override, fallback,
  `ComplianceDecision` construction — is untouched) plus a `votes` breakdown.
- **LLM:** uses `call_llm()` only (provider-agnostic). Extra samples pass a non-zero
  `temperature` and distinct `seed`s (call_llm already supports `seed`) so the votes
  actually differ and remain reproducible.
- **Config:** `settings.platform.self_consistency` (new typed block).
- **Out:** `ComplianceDecision` gains an optional `vote_breakdown: dict = {}` (defaulted →
  backward-compatible) for audit; downstream consumers ignore it unless interested.

### Algorithm
```
parsed_0 = call_llm(temperature=0, seed=0)          # deterministic baseline
if not cfg.enabled or conf(parsed_0) not in [min,max]:
    return parsed_0, {samples: 1}
samples = [parsed_0] + [call_llm(temperature=cfg.temperature, seed=i)
                        for i in 1..cfg.samples-1]
tally   = count of each parsed.decision across samples
winner  = decision with strict majority (> samples/2)
if no strict majority:                              # owner decision: fail-safe
    winner = "insufficient_evidence"
representative = highest-confidence sample whose decision == winner
                (synthetic insufficient_evidence record if winner came from the tie rule)
representative.confidence = winning_votes / len(samples)   # agreement ratio
return representative, {samples, tally, winner}
```
Borderline band is **on the primary confidence**, evaluated once — no chicken-and-egg.

- **Integration:** run `_evaluate_mandatory_check` end-to-end with a borderline-confidence
  mocked primary and divergent samples → correct `ComplianceDecision` + `vote_breakdown`.

## Risks / notes
- **Diversity needs temperature > 0.** At temp 0 the 3 samples are identical and voting is
  a no-op. The first call stays deterministic (audit baseline); only resamples use
  `temperature`. Surface this in the yaml comment so nobody "fixes" it back to 0.
- **Cost.** Bounded: extra calls fire only inside the borderline band, a minority of checks.
  Worst case = `samples`× on those checks. Log the count of voted checks per run.
- **Three-way ties are real** (pass/fail/insufficient are 3 outcomes). With `samples=3` a
  1/1/1 split has no majority → fail-safe `insufficient_evidence` (owner decision; matches
  E3.b "can't confirm → insufficient"). Even `samples` could 2/2 tie — recommend odd.
- **Determinism for tests:** fixed `seed`s + mocked `call_llm` keep unit tests deterministic;
  the benchmark tolerates the small temperature because borderline checks are rare on the
  clean synthetic set (verify no flips).
