## Context / problem

The API serves **58 routes across 8 routers** ([app/api/](../../app/api/)), auto-documented
by FastAPI's Swagger UI at `/docs` and ReDoc at `/redoc`. Today that auto-doc is thin:

- Every router already sets `prefix` + `tags` (grouping works) and the app sets
  `title` + `version` ‚Äî so the skeleton is there.
- **But:** only **13 / 58** routes declare a `response_model`; most routes have **no
  `summary`**, no human description (docstring), no documented **error responses**
  (`401/403/404/409/422`), and several return a raw `dict` so Swagger shows an empty
  `{}` schema. The app itself has no `description`, contact, license, or per-tag
  descriptions.

The result: a buyer / integrating engineer opening `/docs` sees operation names derived
from function names and mostly-empty response schemas ‚Äî below the bar for a
customer-facing enterprise API. This is the DX-001 quick win: **mechanical, customer-
facing, low-risk** ‚Äî make `/docs` self-explanatory without changing any behaviour.

### Benchmark to 2026
FastAPI best practice (and what Stripe/Twilio-grade API docs show): every operation has a
`summary` + description, a typed `response_model`, documented error responses with
examples, tag-level descriptions, and app-level `description`/contact/license. All of this
is **declarative metadata** that feeds the OpenAPI 3.1 schema FastAPI already generates ‚Äî
no new dependency, no runtime cost.

---

## The one real risk ‚Äî `response_model` filters the response

Adding `response_model=X` to a route that returns a **raw dict** makes FastAPI **validate
and FILTER** the response down to `X`'s fields ‚Äî silently dropping any extra keys and
coercing types. On the ~45 un-modelled routes (many return ad-hoc dicts), a careless
`response_model` is a **behaviour change**, not an annotation.

**Rule for this task:** `summary`, `description`, `tags`, `status_code`, and `responses=`
(error docs/examples) are **pure metadata ‚Äî zero runtime change**, apply everywhere.
`response_model` is applied ONLY where (a) a Pydantic model already exists and the handler
already returns exactly that shape, or (b) we add `response_model_exclude_none`/an accurate
model AND prove the response body is byte-unchanged by the regression suite. When in doubt,
document the shape via `responses={200: {"content": {"application/json": {"example": ...}}}}`
instead of forcing `response_model`.

## Per-feature contract

### Exit criteria (testable)
1. **Every** one of the 58 routes has a non-empty `summary` and a description (docstring
   or `description=`). *(meta-test over `app.openapi()` ‚Äî fails if any operation lacks
   `summary`; this gate also prevents future routes from regressing.)*
2. Every route declares an explicit `status_code` (or inherits the correct FastAPI default
   for its method) and documents its **error responses** (`responses={...}`) for each
   `HTTPException` it can raise (401/403/404/409/422 as applicable). *(meta-test: each
   route that has auth deps documents 401/403; each `/{id}` route documents 404.)*
3. `response_model` is set on routes whose handler already returns a typed model verbatim;
   **no route's JSON response body changes** as a result. *(regression: existing
   `tests/regression/` + a before/after response-snapshot test on a representative set.)*
4. App-level OpenAPI metadata enriched: `description`, `contact`, `license_info`, and
   `openapi_tags` with one-line descriptions per tag. *(unit: assert present in
   `app.openapi()["info"]` and `["tags"]`.)*
5. `/docs`, `/redoc`, and `/openapi.json` load with **zero schema-generation errors**
   (`app.openapi()` does not raise; the schema validates as OpenAPI 3.1). *(unit.)*
6. **No behaviour change anywhere**: same routes, same status codes, same bodies, same
   auth. Full non-live suite green; benchmark untouched (no agent code touched). *(suite +
   benchmark.)*

### Module-interaction map
- **Touched:** the 8 route files under [app/api/](../../app/api/) (decorator kwargs +
  docstrings only ‚Äî **no handler-body logic changes**) and [app/main.py](../../app/main.py)
  (`FastAPI(...)` metadata + `openapi_tags`).
- **In/out contracts:** unchanged. `response_model` references existing models in
  [app/schemas/](../../app/schemas/); no new request/response shapes invented. If a route
  genuinely needs a response model, add a typed one to `app/schemas/` rather than inlining.
- **No DB, no agents, no providers touched.** This is an API-surface/documentation layer
  change only.

### Decisions to confirm (at session start)
- **A. `response_model` aggressiveness** ‚Äî *Recommended:* conservative (only where a model
  already matches; document the rest via `responses` examples). Alternative: author
  accurate response models for all 45 un-modelled routes (bigger, higher behaviour-change
  risk ‚Äî likely a second session).
- **B. Meta-test strictness** ‚Äî *Recommended:* hard-fail CI if any route lacks a `summary`
  (criterion #1), so new routes can't regress the docs. Alternative: warn-only.
- **C. Contact/license values** ‚Äî pull from config/`.env` (`platform.yaml` or a new
  `api_docs` block) rather than hardcoding name/email/license in `main.py` (no hardcoding
  rule). Confirm the values to use.

### Test plan
- **Unit (`tests/test_openapi_annotations.py`, new):**
  - completeness meta-test: iterate `app.openapi()["paths"]`, assert every operation has a
    non-empty `summary` and `description` (criterion #1).
  - error-response meta-test: routes with auth deps document 401/403; `/{id}` routes
    document 404 (criterion #2).
  - app-info test: `description`/`contact`/`license_info`/per-tag descriptions present
    (criterion #4); `app.openapi()` runs without raising and reports `openapi: 3.1.x`
    (criterion #5).
- **Regression (response-snapshot):** for a representative route per router, capture the
  JSON body before and after and assert equality (criterion #3 ‚Äî `response_model` didn't
  strip/coerce anything). Reuse `tests/regression/` harness + `fastapi.testclient`.
- **Full non-live suite + benchmark:** green / unchanged (criterion #6).

## Risks / notes
- **Pure-metadata is safe; `response_model` is the only footgun** ‚Äî see the dedicated
  section. Keep #3's snapshot test honest.
- **Scope size:** 58 routes is a lot of decorators but genuinely mechanical. If decision A
  goes "model everything", split into two sessions (metadata first, response models
  second) to respect one-subtask-per-session.
- **Auth/tenant routes** ‚Äî documenting 401/403 must match the *actual* dependency on each
  route (some are public: `/auth/token`, health). Don't blanket-document 401 on public
  routes. Read each route's `Depends(...)` before annotating.
- **`operation_id`** ‚Äî FastAPI derives ugly `operation_id`s from function+path. Optional
  nicety: set clean `operation_id=` for SDK-generator friendliness; out of scope unless
  cheap.

## What was built (2026-06-06)

**Outcome:** all **58 routes + `/health` = 59 documented operations** now declare a
`summary` and a human description; authenticated routes document `401`; path-param
routes document `404`/`409`; role-gated routes document `403`; app-level OpenAPI
metadata (description, contact, license) + per-tag descriptions are populated from
config. `/docs`, `/redoc`, `/openapi.json` generate as OpenAPI **3.1.0** with no errors.

**Config (no hardcoding, decision C):**
- `app/config/loader.py` ‚Äî new `PlatformApiDocs` + `PlatformApiDocsTag` models;
  `PlatformConfig.api_docs` defaulted to `PlatformApiDocs()` so an older
  `platform.yaml` still loads.
- `app/config/platform.yaml` ‚Äî new `api_docs` block (description, contact name/email/url,
  license, 9 tag descriptions). Edit here to rebrand the docs without touching code.

**App wiring:**
- `app/main.py` ‚Äî `_openapi_metadata()` builds `description`/`contact`/`license_info`/
  `openapi_tags` from `settings.platform.api_docs` (empty values omitted). `/metrics` is
  now `include_in_schema=False` (Prometheus scrape endpoint, not customer API; endpoint
  itself unchanged). `/health` gained `tags=["system"]` + summary + docstring.

**Reusable helper:** `app/api/openapi_responses.py` ‚Äî error-spec constants
(`UNAUTHORIZED`/`FORBIDDEN`/`NOT_FOUND`/`CONFLICT`/`BAD_REQUEST`/`UNPROCESSABLE`/
`SERVICE_UNAVAILABLE`/`PAYLOAD_TOO_LARGE`) each documenting the app's `{"detail": str}`
envelope, plus a `responses(*specs)` merge fn. Keeps the 58 decorators DRY.

**Routes annotated (decorator kwargs + docstrings only, no handler-body changes):**
auth (9), admin (9), evaluate (22), tenant (5), org-settings (3), chat (3), logs (3),
rfps (4). Each error code is attached ONLY where the handler can actually raise it
(read every `Depends`/`HTTPException` first ‚Äî public routes are not given a false 401).

**`response_model` (decision A ‚Äî conservative):** NO new `response_model` was added.
The existing 13 explicit `response_model=` values were reformatted onto their own lines
(same value); 6 more routes bind a model via a pre-existing `-> dict`/`-> Model` return
annotation. The snapshot guard pins all 19. ‚áí no route's JSON body can be filtered.

**Tests (new):**
- `tests/test_openapi_annotations.py` ‚Äî meta-tests over `app.openapi()` + the route
  table: completeness (summary+description, **hard-fail**, decision B); authenticated
  routes (detected by walking each route's `dependant` tree for `get_current_user`)
  document 401; path-param routes document 404/409; error examples use the `{"detail"}`
  envelope; app-info + per-tag descriptions present; schema is 3.1; and
  `RESPONSE_MODEL_BASELINE` (19) is unchanged (anti-footgun guard).
- `tests/test_openapi_response_snapshot.py` ‚Äî live body snapshot via `TestClient` WITHOUT
  the lifespan context (no DB): `/health` body byte-equal, `/metrics` served-but-unschemaed,
  an authed route returns the documented `{"detail"}` 401 envelope.

**Verification:** 11 new tests pass; full non-live suite **366 passed, 1 skipped**
(the `RUN_LIVE_LLM` test), 0 failures. Benchmark NOT re-run live ‚Äî this change touches
no agent/retrieval/pipeline code, and the full suite proves settings load with the new
config, so benchmark output is unaffected by construction.

**403 note:** the meta-test does NOT auto-require 403 (most role checks live in handler
bodies, not dependencies, so it isn't reliably detectable). 403 is annotated by hand
per-route where the handler/dependency raises it.

**`operation_id`** (SDK-friendly names): left as out-of-scope per the plan ‚Äî not cheap.
